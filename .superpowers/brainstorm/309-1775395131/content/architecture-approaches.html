Architecture Approaches

Three ways to structure the PM Dashboard — pick one or mix elements

A

Recommended: Next.js App Router + API Routes + OpenClaw Gateway

How it works: Next.js handles everything — pages, API routes that proxy to GitHub/OpenClaw, and server components that fetch data. Your API routes act as a thin middleware between the dashboard and external services.

Pros

  • Single deployable — one Next.js app does it all
  • Server components fetch GitHub data without exposing tokens to the browser
  • API routes proxy OpenClaw calls, keeping your VPS address private
  • Simplest to build, deploy, and maintain
  • shadcn/ui works natively with App Router
  • Vercel AI SDK integrates directly into API routes

Cons

  • All logic in one app — could get large over time
  • API routes have cold start on serverless (Vercel), fine on self-hosted
  • WebSocket support requires custom server if on Vercel

Best for: Solo developer, fast iteration, simplest path to production

B

Next.js Frontend + Separate Backend API

How it works: Next.js is purely the UI. A separate Express/Fastify backend handles GitHub API calls, OpenClaw communication, auth, and data aggregation. Frontend calls your backend, backend calls external services.

Pros

  • Clean separation of concerns — UI vs data/logic
  • Backend can maintain persistent WebSocket connections to OpenClaw
  • Easier to scale independently (frontend CDN, backend horizontally)
  • Backend can cache GitHub data and manage rate limits centrally

Cons

  • Two services to build, deploy, and maintain
  • Extra network hop for every request
  • More complex local development setup
  • CORS configuration needed
  • Overkill for a single-user dashboard

Best for: Teams, complex backends, or if you expect heavy backend logic

C

Next.js + BFF (Backend-for-Frontend) on the Same Server

How it works: Next.js with a custom server (Express wrapper). The custom server handles WebSocket connections for real-time chat, while Next.js API routes handle REST calls. Deployed as one process but with full server capabilities.

Pros

  • Single deployment like Approach A, but with full WebSocket support
  • Can maintain persistent connections to OpenClaw
  • More flexibility than pure Next.js API routes

Cons

  • Custom server disables some Next.js optimizations (automatic static optimization)
  • More complex setup than vanilla Next.js
  • Can't deploy on Vercel — needs a VPS or container
  • SSE streaming (used by Vercel AI SDK) works without WebSocket anyway

Best for: If you need true WebSocket for features beyond chat streaming
